# PSO-Warm-Started Multimodal LLM Agent for Wildfire Spread Prediction

This repository implements a research MVP that uses a lightweight PSO/physics
forecast as a warm start, then lets a multimodal LLM-style reasoner produce
structured correction operations that are rendered by deterministic tools and
checked by guardrails. A learned pixel-level calibrator combines the physical
and LLM-derived channels into a final operational burn-probability map.

The implementation is model-pluggable: it ships with a deterministic
heuristic reasoner so the pipeline, metrics, and ablations can run without any
LLM serving infrastructure, and it can be pointed at a local Ollama model
(e.g. `llama4:latest`, `gemma4:31b`) for the full multimodal-LLM condition.

## Research Position

Do not claim that an off-the-shelf LLM directly simulates wildfire physics.
The defensible claim is:

> Explicit physical priors and constrained tool rendering allow a multimodal
> LLM agent to improve PSO/physics warm-start forecasts, and a learned
> pixel-level calibrator improves on both, without domain-specific LLM
> training.

LLM-derived channels should be described as auxiliary structured evidence for
new-spread detection, not as the dominant source of accuracy or as a
standalone spread predictor — the no-LLM ablation (`calibrated_no_llm` in the
CLI, see below) is the direct check for this.

## Repository Scope

This repository tracks **experiment code only**: `src/`, `scripts/`,
`tests/`, and this README. It does not track:

- `dataset/` (large materialized raster/weather data — versioned separately),
- `outputs/` (all experiment results — regenerated by running the CLI, see
  below),
- `docs/` and `paper/` (paper-writing notes, reviewer/audit trail, and the
  LaTeX manuscript — a separate, non-code deliverable).

All of the above are listed in `.gitignore`. If you have a local checkout with
those directories already populated, they remain on disk; they are simply not
part of this repository going forward. This README is the single source of
truth for reproducing the code's results from scratch.

## Setup

```powershell
python -m venv .venv
.venv\Scripts\pip install -e .
# Optional: GeoTIFF reading for materialized static layers (Fuel_Map, Topography_Map)
.venv\Scripts\pip install -e .[geo]
# Optional: test runner extras
.venv\Scripts\pip install -e .[dev]
```

Requires Python 3.10+. Core dependencies are `numpy` and `pillow`; the
synthetic demo and unit tests do not require `rasterio` or any LLM serving
stack.

Two external dependencies are optional and only needed for the full
experimental pipeline (not for the synthetic demo or unit tests):

- **PSO/physics warm start**: the CLI imports an external `ros_wildfire`
  package (Huygens/PSO wildfire simulator). Point `--ros-wildfire-src` at the
  local `ros-based-wildfire-prediction/src` checkout if it is not importable
  from the active environment. Use `--warm-start placeholder` to skip PSO
  entirely for quick debugging. Run `python -m wildfire_llm_agent.cli doctor`
  to check whether PSO imports cleanly and whether the `.venv` user-site
  `torch` package is visible.
- **LLM reasoner**: pass `--reasoner ollama --ollama-model <name>` to use a
  local Ollama-served multimodal model (e.g. `llama4:latest`,
  `gemma4:31b`). Without `--reasoner ollama`, the CLI falls back to a
  deterministic heuristic reasoner, so the full pipeline still runs
  end-to-end without any model serving.

## Pipeline Architecture

```
burn_map_t, weather_t, static layers
        |
        v
  PSO / physics warm-start forecast  ---->  warm_start
        |
        v
  context builder + rule-based RAG retriever (fire-behavior snippets)
        |
        v
  LLM reasoner (heuristic or Ollama)  ---->  structured correction plan (JSON)
        |
        v
  deterministic tool renderer (dilation, downwind expansion, fuel mask,
  humidity suppression, monotonic-burn constraint)
        |
        v
  guardrails (shape/probability/operation validation)  ---->  agent, llm_only
        |
        v
  area-budget selector (reject over-expansive plans, fall back to warm_start)
        ---->  agent_selected
        |
        v
  learned pixel-level calibrator (logistic regression over warm/agent/
  llm_only/simple-physics probabilities + burn-front/wind/weather/fuel/
  terrain features)  ---->  calibrated_raw, calibrated
```

Evaluation is strictly one-step-ahead: every method predicts
`burn_map_{t+1}` from `burn_map_t` and other data available at time `t` only.

## Method Name Glossary

These are the `method` values that appear in every `metrics.csv`:

| Method | Meaning |
| --- | --- |
| `persistence` | current burn map copied forward (naive baseline) |
| `warm_start` | physical-only PSO/physics forecast |
| `simple_physics` | deterministic non-PSO heuristic spread baseline |
| `llm_only` | LLM/RAG/tool-rendering pipeline without the PSO forecast; base map is `burn_map_t` |
| `agent` | raw hybrid: PSO warm start + LLM correction, before selection |
| `agent_selected` | guarded hybrid after the area-budget selector |
| `calibrated_raw` | learned pixel-level new-burn probability map, before threshold/area post-processing (use for AUPRC/ranking analysis) |
| `calibrated` | learned pixel-level hybrid after train-fit threshold + prior-based area budget (the strongest operational predictor) |
| `calibrated_no_llm` | same calibrator, fit without any LLM-derived feature (the no-LLM ablation) |
| `supervised_pixel_logistic` | same-dataset supervised reference baseline (logistic regression directly on `t+1` labels, no physical/LLM structure) |
| `random_50` | destructive sanity baseline: each eligible pixel flips to newly-burned with p=0.5 |
| `random_warm_area` | sanity baseline: random pixels matching the warm start's predicted new-burn area |
| `random_oracle_prevalence` | non-deployable oracle sanity baseline: random pixels matching the true new-burn count |

## Core Evaluation Workflow

Replace `<ROS_WILDFIRE_SRC>` with the local path to the external
`ros-based-wildfire-prediction/src` checkout in every command below (or omit
`--ros-wildfire-src` if `ros_wildfire` is already importable).

Run a smoke evaluation on the materialized dataset root. Use `--pso-t-cap 48`
for any result you intend to report; smaller values are debugging shortcuts
only:

```powershell
python -m wildfire_llm_agent.cli evaluate-materialized-root --dataset-root dataset\pkl_materialized --max-scenarios 1 --max-events-per-scenario 1 --max-frame-pairs-per-event 2 --output-dir outputs\materialized_eval_pso_smoke --warm-start pso --pso-device cpu --pso-t-cap 48 --ros-wildfire-src <ROS_WILDFIRE_SRC> --save-panels
```

Run the physical-only vs. LLM-only vs. hybrid ablation with the area-budget
selector and Llama4 through Ollama:

```powershell
python -m wildfire_llm_agent.cli evaluate-materialized-root --dataset-root dataset\pkl_materialized --max-scenarios 2 --max-events-per-scenario 2 --max-frame-pairs-per-event 3 --output-dir outputs\ablation_hybrid_prior_gated_ratio10 --warm-start pso --pso-device cpu --pso-t-cap 48 --ros-wildfire-src <ROS_WILDFIRE_SRC> --pso-cache-dir outputs\pso_cache --reasoner ollama --ollama-model llama4:latest --ollama-timeout-seconds 600 --selector area_budget --selector-max-new-to-current-ratio 10 --selector-prior-fallback llm_only --include-llm-only --ollama-cache-dir outputs\ollama_cache
```

Run the pixel-level calibration experiment (trains the learned calibrator on
early events per scenario, evaluates on a held-out event):

```powershell
python -m wildfire_llm_agent.cli calibrate-materialized-root --dataset-root dataset\pkl_materialized --max-scenarios 5 --max-events-per-scenario 3 --train-events-per-scenario 2 --max-frame-pairs-per-event 5 --output-dir outputs\pixel_calibration_5scenario_5frames --warm-start pso --pso-device cpu --pso-t-cap 48 --ros-wildfire-src <ROS_WILDFIRE_SRC> --pso-cache-dir outputs\pso_cache --reasoner ollama --ollama-model llama4:latest --ollama-timeout-seconds 600 --selector area_budget --selector-max-new-to-current-ratio 10 --selector-prior-fallback llm_only --ollama-cache-dir outputs\ollama_cache --calibration-iterations 600 --calibration-learning-rate 0.08
```

For a larger robustness check, rerun the same command with `--max-scenarios
10` and `--output-dir outputs\pixel_calibration_10scenario_5frames`.

Keep `--pso-cache-dir outputs\pso_cache` and `--ollama-cache-dir
outputs\ollama_cache` fixed across runs to avoid repeated PSO simulation and
LLM calls.

Run rotating held-out-event calibration to check split sensitivity:

```powershell
python -m wildfire_llm_agent.cli cross-validate-materialized-calibration --dataset-root dataset\pkl_materialized --max-scenarios 5 --max-events-per-scenario 3 --max-frame-pairs-per-event 5 --output-dir outputs\pixel_calibration_rotating_cv_5scenario_5frames --warm-start pso --pso-device cpu --pso-t-cap 48 --ros-wildfire-src <ROS_WILDFIRE_SRC> --pso-cache-dir outputs\pso_cache --reasoner ollama --ollama-model llama4:latest --ollama-timeout-seconds 600 --selector area_budget --selector-max-new-to-current-ratio 10 --selector-prior-fallback llm_only --ollama-cache-dir outputs\ollama_cache --calibration-iterations 600 --calibration-learning-rate 0.08 --min-train-events 2
```

## Experiment Protocol

**Objective**: measure whether a PSO/physics warm-start forecast can be
improved by a multimodal LLM-style correction agent, without domain-specific
LLM training. The core temporal hypothesis is that the PSO warm start
stabilizes early predictions, while LLM correction becomes more useful as the
fire evolves and richer burn-front evidence accumulates.

**Required inputs per prediction**: `burn_map_t`, `pso_forecast_t_plus_h`,
static layers (aspect, elevation, slope, FBFM13, canopy cover), `weather_t`,
and a held-out `target_burn_map_t_plus_h` for evaluation only.

**Baselines**: persistence, PSO/physics-only, prompt-only (no RAG), PSO+RAG
without tool rendering, PSO+RAG+structured tool rendering, no-LLM
calibration, supervised pixel-logistic reference, and three random sanity
baselines (see the glossary above).

**Primary metrics**: newly burned F1 and IoU (primary — full-map IoU/F1 can
be inflated by persistence when spread is slow), newly burned AUPRC,
precision, recall, burned-area relative error, boundary F1, wind-aligned
spread error, and latency per prediction.

**Leakage controls**:
- Split by fire event first; do not let adjacent frames from the same event
  cross calibration/evaluation sets.
- Each one-step prediction uses only data available at time `t`
  (`burn_map_t`, `weather_t`); the ground-truth `burn_map_{t+1}` is never fed
  to the PSO forecast, LLM prompt, retrieval, renderer, guardrails, selector,
  or calibrator.
- Static layers (terrain, fuel) are reused across timesteps because they are
  time-invariant in this dataset.
- Thresholds and area budgets for the calibrator are fit only on calibration
  events; rotating cross-validation refits them after holding out each event.

**Standardization**: use `--pso-t-cap 48` for any run whose numbers you
intend to report; smaller caps are debugging shortcuts and change PSO
simulation depth.

## Model Selection Notes

- `llama4:latest` (Ollama-served, ~108B params, vision+tools) is the primary
  multimodal reasoner: it is natively multimodal and matches the project's
  technical claim of high-capacity multimodal reasoning over a physical warm
  start.
- `gemma4:31b` (~31B params, vision+tools+thinking) is a smaller, faster
  efficiency/robustness comparison model, not the primary condition. In
  small-sample smoke tests it behaved as a more conservative, lower-variance
  planner.
- Both models are slow enough (tens of seconds to minutes per call on local
  hardware) that large batch runs require `--ollama-cache-dir` reuse across
  runs; do not interpret cached-run latency as live inference latency.

## Known Limitations

- The full materialized dataset used for development has only 10 fire
  scenarios; held-out evaluations in the calibration commands above typically
  cover single-digit numbers of held-out events. Treat any percentage-style
  improvement claims as based on a small sample, and prefer the event-level
  bootstrap intervals (`--unit event`) over point estimates.
- PSO frame sequences can be incomplete for some events; `calibrate-materialized-root`
  and `cross-validate-materialized-calibration` use an adaptive per-event
  `--pso-t-cap` fallback so incomplete sequences are still evaluated over
  their available horizon rather than skipped or padded with synthetic data.
- The correction renderer can over-expand small fires; the area-budget
  selector (`--selector area_budget`) exists specifically to catch and reject
  this failure mode, falling back to `warm_start` or `llm_only`.
- Large batch commands (many scenarios/events with a live Ollama model) can
  take a long time on cold cache; always set `--ollama-cache-dir` and
  `--pso-cache-dir` and expect the first run in a new cache directory to be
  the slow one.
